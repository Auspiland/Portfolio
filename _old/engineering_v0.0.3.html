00 / 일하는 방식

진단보다 먼저, 태도

운영 시스템을 다루는 일은 "고치는 것"보다 "함부로 건드리지 않는 것"에서 출발합니다. 네 사례 모두를 관통하는 세 가지 원칙입니다.

01

read-only 우선

운영·개발 서버는 승인 없이 쓰지 않습니다. 진단은 get·cat·describe·SELECT 같은 읽기로만 끝내고, 변경은 반드시 확인 후에.

02

다중 증거 수렴

단일 신호로 단정하지 않습니다. mtime·파일권한·md5·디컴파일·DB row가 한 방향을 가리킬 때 비로소 결론을 못 박습니다.

03

가설 → 검증 → 정정

매끄러운 추론으로 빈틈을 메우지 않습니다. 코드로 확인한 사실과 정황을 구분하고, 틀린 가설은 보고서에 정정 노트로 남깁니다.

01 / GraphRAG 인덱싱

비용을 아는 설계

Design · GraphRAG

GraphRAG 증분 인덱싱

CASE 01

문서가 계속 쌓이는 GraphRAG 인덱스 — 새 문서 몇 건 때문에 LLM 비용을 들인 전체 그래프를 매번 다시 만들 순 없었습니다.

진짜 원인 · 해법

핵심은 "비싼 건 보존, 새 것만 발급". 제목을 정규화해 기존 개체의 ID를 그대로 물려주고 매칭 안 되는 것만 새로 만들면, 전체 재빌드 없이 새 문서만 그래프에 싸게 덧붙는다.

S상황

개체·관계·커뮤니티 리포트를 전부 LLM으로 생성하는 구조라, 문서가 추가될 때마다 전체 재빌드를 도는 건 비용·시간상 현실적으로 불가능했습니다.

T과제

LLM이 이미 돈을 들여 만든 자산(개체 ID·요약·리포트)을 깨뜨리지 않으면서, 새 문서만 그래프에 덧붙이는 방식을 설계하고 구현하기.

A행동

세 가지로 풀었습니다. ① 개체 — 정규화한 제목으로 이전 실행의 ID를 찾아 그대로 승계하고, 못 찾은 것만 새 ID 발급 → 참조가 끊기지 않음. ② 커뮤니티 — Seeded Leiden + 멤버 겹침 1:1 매칭으로 군집 ID를 안정적으로 이어받음. ③ LLM 재계산 — 변화량이 임계 미만이면 요약·리포트 재생성을 건너뜀. 단, 정규화 규칙 자체가 바뀌면 승계가 깨지므로 canonicalization_version이 바뀔 때만 전체 재정합을 강제하는 안전장치를 뒀습니다.

R결과

ID 재발급 0건으로 LanceDB·OpenSearch·서빙의 참조 붕괴를 원천 차단하고, 증분 실행만으로 그래프를 갱신. 전체 재빌드 대비 LLM 재계산을 약 87% 줄였습니다. (128개 테스트 통과로 검증, 구현 중 설계 사각을 스스로 발견해 문서를 v1.1로 정정.)

02 / 레거시 CMS

URL 한 줄에서 시스템 전체로

Reverse Engineering

CMS ↔ t3q AI 아키텍처 역추적

CASE 02

인수인계 문서가 한 장도 없는 레거시 미디어 CMS — 가진 단서는 브라우저 개발자도구에 찍힌 요청 URL 한 줄뿐이었습니다.

진짜 원인 · 발견

외부 SaaS인 줄 알았던 api.kpf-dev.t3q.ai가, 알고 보니 같은 쿠버네티스 클러스터 안에서 도는 자체 제작 FastAPI 서비스였다 — 컨테이너에 직접 들어가 라우터 코드를 읽고 확정.

S상황

운영 중인 CMS인데 매뉴얼이 없고, 호출하는 외부 API의 서버측 실체가 외부 SaaS인지 내부 서비스인지조차 알 수 없었습니다.

T과제

업로드 → 표지 자동생성 → 검색 색인까지 전 과정을, "어디까지가 외부이고 누가 호출하는가"를 추측이 아니라 코드 근거로 확정하기.

A행동

요청 URL 문자열을 코드에서 grep해 컨트롤러 → 매퍼 → 테이블 → NAS 경로로 한 단계씩 따라 내려갔습니다. 외부처럼 보이던 도메인은 kubectl로 파드 안에 들어가 FastAPI 라우터를 직접 읽어 내부 서비스임을 확정. 표지를 만든 주체도 디렉터리 이름(algo_9)에 속지 않고 서빙 이미지 → 파드 로그 → 호출자 IP 역조회 → NiFi 설정까지 교차검증해, 처음 세운 가설을 뒤집고 실제 주체를 바로잡았습니다.

R결과

CMS → 내부 검색 서비스 → OpenSearch 흐름과 NiFi·Spark 경로를 하나의 End-to-End 문서로 통합. 외부 API 13종을 호출 8·미호출 4·콜백 2로 분류해 인벤토리화하고, 표지 누락 시 진단 기준(시나리오 매트릭스)을 만들어 운영 기준선을 세웠습니다.

데이터레이크 누락 추적

CASE 03

17개로 쪼갠 잡지 중 특정 청크만 CMS 검색에 안 잡힘 — 데이터가 깨진 건지, 처리 흐름이 막힌 건지 알 수 없었습니다.

진짜 원인 · 규명

데이터 결함이 아니었다. 표지 JPG 레코드가 생겨야만 색인이 진행되는 구조(= 색인의 관문)인데, 그 레코드가 일부 청크에만 만들어졌던 것.

S상황

17개 청크로 분할된 잡지 중 일부가 CMS 검색에서 누락. 원본이 깨진 건지, 처리 흐름이 막힌 건지 불명확했습니다.

T과제

파일 실체(NAS·HDFS)와 메타데이터(OpenSearch·DB) 양쪽에서 누락이 "어디서 처음 끊겼는지"와 원인을 규명하기. (모든 원격 작업은 read-only 원칙 준수)

A행동

NAS·HDFS·OpenSearch·PostgreSQL 네 레이어를 같은 17청크 기준으로 나란히 대조했습니다. PDF 17개는 NAS에 모두 정상 적재됐지만 HDFS·색인·DB로 갈수록 일부만 남았고, DB(t_comm_file_inf)를 read-only로 조회하니 표지 JPG 레코드가 누락된 청크에만 없었고 갱신 시각도 뒤늦게 찍혀 있었습니다. 운영 Spark가 OOM(exit 137)으로 막던 집계는 로컬 DuckDB로 우회해 끝냈습니다.

R결과

"표지 JPG 레코드 생성이 색인의 관문인데 일부만 성공했기 때문"임을 4레이어+DB 증거로 확정. JPG 레코드를 넣은 코드는 디컴파일로 확인하지 못해, 단정 대신 "정황"으로 명시했습니다. 부수로 72만 파일을 인벤토리화해 "표지=JPG / PNG=HWP 섹션" 규칙을 정리했습니다.

04 / Spark 파이프라인

레이어를 격리해, 한 줄까지

Debugging · Ops

Spark 파이프라인 부활

CASE 04

의도적으로 꺼져 있던 스캔 PDF 청킹 3단 Spark 라인 — "yaml 하나만 넣으면 되살아나냐"는 검토 요청에서 시작했습니다.

진짜 원인 · 처치

코드 버그가 아니었다. chunk 단계가 바라보던 추론 서버 주소가 HDFS 권한 없는 인스턴스(operservice-1)를 가리켰고, 그게 돌려준 에러 응답을 클라이언트가 리스트처럼 순회하다 터진 것. 답은 yaml의 주소를 operservice-2로 바꾸는 한 줄이었다.

S상황

batch→chunk→indexing 3단 Spark 라인이 비활성 상태. "yaml 하나만 추가하면 되살아나나?"라는 검토 요청이 들어왔습니다.

T과제

부활 가능 여부와 스키마 정합을 검증하고, 가능하면 실제 데이터를 흘려 정상화하기 — 단, 운영은 멈추지 않게.

A행동

세 단계가 batch_start_page 컬럼으로 묶인 불가분 세트임을 확인하고 순차 적용으로 cold-start를 우회. chunk 단계에서 터지던 RuntimeError를 잡으려 살아있는 driver 파드 안에서 추론 엔드포인트를 직접 POST로 찔러보니, operservice-1이 "HDFS 접근 실패" 응답(dict)을 돌려주고 클라이언트가 그걸 리스트처럼 순회하다 폭주함을 코드 라인까지 추적. operservice-1과 2가 같은 이미지의 다른 인스턴스인데 1에만 HDFS 환경변수가 없다는 차이를 찾아냈습니다.

R결과

진짜 원인은 코드가 아니라 추론 서버 주소를 operservice-1→2로 바꾸는 yaml 한 줄. 모든 부활 시도는 롤백해 운영을 시도 전과 100% 동일하게 복구했습니다. (부수: Spark Operator "pod 안 뜸"을 단위 표기 + JSON 오타 2단 원인으로 분해, Helm 무중단 증설, 멱등·안전가드 기반 대용량 이관.)
