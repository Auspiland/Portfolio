00 / 일하는 방식

진단보다 먼저, 태도

운영 시스템을 다루는 일은 "고치는 것"보다 "함부로 건드리지 않는 것"에서 출발합니다. 네 사례 모두를 관통하는 세 가지 원칙입니다.

01

read-only 우선

운영·개발 서버는 승인 없이 쓰지 않습니다. 진단은 get·cat·describe·SELECT 같은 읽기로만 끝내고, 변경은 반드시 확인 후에.

02

다중 증거 수렴

단일 신호로 단정하지 않습니다. mtime·파일권한·md5·디컴파일·DB row가 한 방향을 가리킬 때 비로소 결론을 못 박습니다.

03

가설 → 검증 → 정정

매끄러운 추론으로 빈틈을 메우지 않습니다. 코드로 확인한 사실과 정황을 구분하고, 틀린 가설은 보고서에 정정 노트로 남깁니다.

GraphRAG 증분 인덱싱

CASE 01

이미 LLM 비용을 들여 만든 지식그래프 인덱스를, 전체 재빌드 없이 새 문서만 싸게 덧붙입니다.

S상황

문서가 계속 추가되는 GraphRAG 인덱스. 개체·관계·커뮤니티 리포트를 LLM으로 생성하는 구조라 전체 재빌드는 비용·시간상 불가능했습니다.

T과제

기존 uuid4 ID와 요약·리포트 자산을 보존하면서 증분만 반영하는 설계를 확정하고 구현하기.

A행동

12개 쟁점을 결정값으로 닫았습니다. 정규화 제목 lookup으로 기존 ID를 승계하고 no-match만 새로 발급, Seeded Leiden + 멤버 overlap 1:1로 커뮤니티 ID를 안정 승계, 변화량 임계(재요약 3 / 리포트 0.2) 기반으로 LLM 재계산을 스킵. 정규화 규칙을 바꾸면 ID 승계가 깨지므로 canonicalization_version 게이트로 변경 시에만 full reconcile을 강제했습니다.

R결과

Phase 1~3 구현 + 128개 테스트 통과. ID 재발급 0으로 LanceDB·OpenSearch·serving의 참조 붕괴 위험을 원천 차단. 구현 중 재검토로 설계 사각(Prev Rows Carry-Over)을 발견해 문서를 v1→v1.1로 동기화했습니다.

Reverse Engineering

CMS ↔ t3q AI 아키텍처 역추적

CASE 02

매뉴얼 없는 레거시 미디어 CMS에서, 브라우저에 찍힌 요청 URL 하나로 다계층 분산 시스템 전체를 코드로 재구성했습니다.

S상황

인수인계 문서가 없는 운영 CMS. 호출하는 외부 API api.kpf-dev.t3q.ai의 서버측 실체가 외부 SaaS인지 내부 서비스인지조차 불명확했습니다.

T과제

업로드 → 표지 자동생성 → 검색 색인까지 전 과정을, 외부/내부 경계와 호출 주체까지 코드 근거로 확정하기.

A행동

요청 URL 문자열을 grep해 컨트롤러 → 매퍼 → 테이블 → NAS 경로로 내려갔습니다. 외부처럼 보이던 도메인은 kubectl exec로 파드 안 FastAPI 라우터를 직접 읽어 같은 클러스터 내부의 in-house t3q-search-svc임을 확정. 썸네일 생성자는 코드 디렉토리명(algo_9)에 속지 않고 serving 이미지 → 파드 로그 → 호출자 IP 역조회 → NiFi flow.xml까지 교차검증해 operservice-15 + NiFi로 못 박고, 초기 가설을 정정했습니다.

R결과

CMS→t3q-search-svc→OpenSearch 흐름과 NiFi·Spark 처리 경로를 단일 End-to-End 문서로 통합. 외부 API 13종을 호출 8·미호출 4·콜백 2로 인벤토리화하고, 표지 placeholder 자동 채움/잔존 조건을 시나리오 매트릭스로 확정해 운영 진단 기준을 만들었습니다.

데이터레이크 누락 추적

CASE 03

HDFS·Delta·NAS에 흩어진 수십만 파일에서, "누가 만들었고 왜 누락됐나"를 여러 레이어의 증거로 규명했습니다.

S상황

17개 청크로 분할된 잡지 중 특정 청크가 CMS 검색에서 잡히지 않았습니다. 데이터 결함인지 처리 흐름 문제인지 불명확.

T과제

파일 실체(NAS·HDFS)와 메타(OpenSearch·DB) 양쪽에서 누락의 정확한 기점과 원인을 규명하기. (모든 원격 작업은 read-only)

A행동

NAS·HDFS·OpenSearch·PostgreSQL 4개 레이어를 교차 대조했습니다. 17개 PDF는 모두 정상·동일 특성인데 HDFS엔 일부만 적재돼 있었고, read-only로 DB(t_comm_file_inf)를 직접 조회하니 표지 JPG row가 그 일부 청크에만 존재하고 JPG row만 갱신 시각이 뒤늦게 찍혀 있었습니다. 운영 Spark가 OOM(exit 137)으로 막히는 통계 작업은 로컬 venv + DuckDB로 우회했습니다.

R결과

"데이터 결함이 아니라 표지 JPG row 생성이 후속 색인의 게이트인데 일부만 성공했기 때문"임을 4-레이어+DB 증거로 확정. JPG row를 INSERT한 코드는 디컴파일 미확인이라 단정 대신 "정황"으로 정정·명시했습니다. (부수: 72만 파일 인벤토리로 "표지=JPG / PNG=HWP 섹션" 규칙화.)

Spark 파이프라인 부활

CASE 04

비활성화된 PDF 청킹 라인을 되살리며, 고장의 진짜 원인을 yaml 한 줄까지 좁혔습니다 — 운영은 멈추지 않고.

S상황

스캔 PDF 청킹 3단 Spark 라인(batch→chunk→indexing)이 의도적으로 비활성화된 상태. "yaml 하나만 추가하면 되살아나나?"라는 검토 요청.

T과제

부활 가능 여부와 스키마 정합을 검증하고, 가능하면 실제 데이터를 흘려 정상화하기.

A행동

세 단계가 batch_start_page 컬럼으로 묶인 불가분 세트임을 입증하고, cold-start는 순차 apply로 우회. 터진 pandas_udf RuntimeError를 잡으려 활성 driver 파드 안에서 inference 엔드포인트를 직접 POST 프로빙했더니, operservice-1이 HDFS 접근 실패 dict를 반환하고 클라이언트가 그걸 list처럼 순회해 폭주함을 코드 라인까지 추적. operservice-1(HDFS env 없음)과 operservice-2(보유)가 같은 이미지의 다른 인스턴스임을 규명했습니다.

R결과

진짜 원인은 코드가 아니라 yaml의 INFERENCE_URL을 operservice-1→2로 바꾸는 한 줄. 모든 부활 시도는 rollback해 운영을 시도 전과 100% 동일하게 복구했습니다. (부수: Spark Operator "pod 안 뜸"을 단위 표기 + JSON 오타 2단 원인으로 분해, Helm 무중단 증설, 멱등·안전가드 14GB 이관.)
