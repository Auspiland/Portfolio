# engineering.html — Placeholder 값 정의서

engineering.html에는 "판을 바꿀 실제 수치"가 들어갈 자리에 **임시 더미값**을 넣고,
HTML 소스에 `<!-- PLACEHOLDER P-XX -->` 주석으로 표시해 두었습니다.
실제 값이 확보되면 아래 정의에 맞춰 더미값을 교체하세요.

> 표기 규칙: HTML metric 박스(`.metric .v`)의 더미값을 실측치로 바꾸고, 주석은 그대로 두거나 제거.

## CASE 01 · GraphRAG 증분 인덱싱

### P-01A — LLM 재계산 절감률 (현재 더미: `↓87%`)
- **의미:** 전체 재빌드 시 호출되는 LLM 요청(개체 추출 + 커뮤니티 리포트 생성) 대비, 증분 실행에서 **실제로 호출된 LLM 요청의 감소율**.
- **측정법:** `1 − (증분 run의 LLM 호출 수 ÷ full rebuild의 LLM 호출 수)`. 동일 코퍼스에 신규 문서 1배치를 추가한 시나리오 기준.
- **왜 중요:** 증분 인덱싱의 존재 이유가 곧 "LLM 비용 절감". 이 한 숫자가 케이스의 가치를 대변함.

### P-01B — 갱신 시간 단축 (현재 더미: `분 단위`)
- **의미:** 신규 문서 1배치를 그래프에 반영하는 데 걸리는 벽시계 시간(full rebuild 대비).
- **측정법:** full rebuild 소요시간 vs 증분 run 소요시간. 예) `4시간 → 6분` 또는 `40배`.
- **왜 중요:** 비용과 별개로 "운영 중 즉시 반영 가능"을 보여줌.

## CASE 03 · 데이터레이크 누락 추적

### P-03A — 근본원인 규명 소요 (현재 더미: `1일 내`)
- **의미:** 누락 신고 접수부터 4레이어 교차대조로 근본원인을 확정하기까지 걸린 시간.
- **측정법:** 조사 시작 ~ 원인 확정 보고까지의 실제 소요(시간/일).
- **왜 중요:** "문서 없는 시스템에서 빠르게 진상에 도달"하는 진단 속도를 정량화.
- (대안 수치: 누락 영향 범위 — 검색에서 빠진 청크/문서 수. 이미 본문에 17청크 중 일부로 정성 표현됨.)

## CASE 04 · Spark 파이프라인 부활

### P-04A — 부활 후 처리량 / 방치 기간 (현재 더미: `정상화`)
- **의미(택1):**
  - (a) 라인이 비활성으로 방치돼 있던 기간 (예: `N개월 비활성`), 또는
  - (b) 부활 후 정상 처리된 문서/페이지 수 또는 처리 속도.
- **측정법:** (a) 비활성 시작 시점 ~ 부활 시점. (b) 부활 직후 1회 배치의 처리 건수.
- **왜 중요:** "한 줄로 되살렸다"의 임팩트를 결과 규모로 뒷받침.
- **참고:** 기존 `14GB 안전 이관` 메트릭은 곁다리 작업이라 메트릭에서 제거하고 본문 부수 언급으로 강등함.
