# Engineering Casebook 재정리 메모 (작업용)

> 목적: engineering.html을 "프로젝트별 트러블슈팅" 관점으로 다시 정리.
> 이 메모는 (1) 강조 수치 검증, (2) 재구성 축, (3) 각 케이스 "해결 과정"이 안 읽히는 이유를 판단하기 위한 스크래치.

## A. 강조 수치(metric) 냉정한 검증

판정 기준: **① 결과(outcome)인가 과정(process)인가  ② 읽는 사람이 의미를 즉시 아는가  ③ 임팩트가 있는가(이목)**
→ 과정-카운트(테스트 수, 보고서 수, 결정 수)는 "열심히 했다"는 신호일 뿐 부풀리기 쉬워 약하다. 살아남아야 할 건 **시스템적 의미가 있는 결과값**.

### CASE 01 · GraphRAG 증분 인덱싱
| 수치 | 판정 | 코멘트 |
|---|---|---|
| `128` Tests Pass | △ 약함 | 테스트 개수는 gameable. "꼼꼼함" 신호지만 결과가 아님. 숫자 자체는 임팩트 적음 |
| `12` 설계 결정 | ✗ 버릴 것 | 가장 약함. "12개 쟁점"은 읽는 사람에게 의미 0. 임의 카운트 |
| `0` ID 재발급 | ◎ 핵심 | **이 케이스의 진짜 성과.** 비싼 LLM 자산(ID) 0개 재발급 = 참조 붕괴 0. 유지 |
| (없음) 비용/시간 절감 | ?? 필요 | 증분 인덱싱의 본질은 "재빌드 대비 LLM 호출/비용 절감". **이 숫자가 있어야 가장 강력.** → 질문 |

### CASE 02 · CMS↔t3q 역추적
| 수치 | 판정 | 코멘트 |
|---|---|---|
| `13` 외부 API | ○ 보통 | 인벤토리 폭을 보여줌. "호출8·미호출4·콜백2" 분해는 좋음. 유지하되 라벨 개선 |

| (없음) "외부로 보였으나 내부였다" | ◎ 스토리 | 숫자는 아니지만 **이 케이스의 하이라이트.** 메트릭보다 본문 헤드라인감 |

| 수치 | 판정 | 코멘트 |
|---|---|---|
| `1줄` 근본 수정 | ◎ 최강 | 전체 장애 → yaml 한 줄. 드라마틱. 유지 (이 페이지 최고의 메트릭) |

### 종합 판정
- **살릴 메트릭:** `0 ID재발급`, `72만 파일`, `1줄 수정`, `100% 롤백`, `5+ 계층`, `13 API`
- **버릴/강등:** `12 설계결정`, `7 보고서`, `128 테스트`(약화), `RO`(배지로), `14GB`(side로)
- **확보하면 판을 바꿀 수치(→ 사용자 질문):**
  - CASE01: 재빌드 대비 **LLM 호출/비용/시간 절감률** 또는 "신규문서 N개만 처리" 같은 비율
  - CASE03: 누락 추적에 걸린 시간, 혹은 영향 범위(검색 누락 문서 수)
  - CASE04: 장애 지속/방치 기간, 부활 후 처리량

> 패턴 진단: 현재 각 케이스가 **강한 스토리 메트릭 1 + 과정 카운트 2** 구성. 과정 카운트를 쳐내고 빈자리는 "본문 강조 문장" 또는 실제 절감 수치로 대체하는 게 핵심.

## B. 재구성 축: "방식" → "프로젝트별 트러블슈팅"

현재는 **스킬 4기둥**(설계/역추적/진단/운영)으로 묶임. 사용자 요청 = **프로젝트(시스템)별 트러블슈팅 서사**로 전환.

각 케이스를 다음 4비트로 다시 읽히게:
```
① 증상(Symptom)   — 무엇이 안 됐나 / 무엇을 요청받았나   (S+T 압축)
② 추적(Investigation) — 어떤 증거를 어떻게 모았나          (A의 전반)
③ 진짜 원인(Root Cause) — 무엇이 범인이었나 (← 지금 가장 안 읽힘)
④ 처치 & 결과(Fix/Result) — 무엇을 했고 뭐가 보장됐나      (A 후반 + R)
```
STAR는 유지하되, **③ 진짜 원인을 시각적으로 분리**해서 "그래서 답이 뭐였는지"가 한눈에 보이게.

## C. "어떻게 해결했나"가 안 읽히는 이유 (케이스별)

공통 병폐: A(행동)가 **고유명사·전문용어를 앞세워** 서술돼서, 정작 "원인→처치"의 인과가 문장 속에 묻힘. 읽는 사람이 `operservice-1`, `batch_start_page`, `canonicalization_version`, `pandas_udf`를 모르면 결론을 못 따라감.

- **CASE01:** lookup으로 ID 승계 / Leiden 커뮤니티 승계 / 임계로 LLM 스킵 — 방법은 있으나 **"그래서 결국 새 문서는 어떻게 싸게 붙는가"**의 한 줄 요약이 없음. 3개 기법이 병렬 나열돼 우선순위 불명.
- **CASE02:** "외부처럼 보인 도메인이 내부 서비스였다"가 최대 반전인데 **A 문장 중간에 묻힘.** kubectl exec로 확정한 절차가 핵심인데 다른 디테일과 동급으로 흐름.
- **CASE03:** 결론("표지 JPG row가 색인의 게이트")은 명확하나, **거기에 도달한 추론 순서**(4레이어를 왜 그 순서로 봤는지)가 안 보임. row 갱신시각이 늦었다는 핵심 단서가 약하게 처리됨.
- **CASE04:** "원인은 yaml 한 줄"은 강하지만, **왜 operservice-1이 문제였나**(HDFS env 없는 인스턴스)가 괄호 디테일로 밀려 인과가 약함. dict를 list로 순회해 폭주 → 이게 진짜 메커니즘인데 한 번 읽고 이해 어려움.

→ 처방: 각 케이스에 **"한 줄 진상(the punchline)"**을 ③로 못박고, A는 그 진상에 이르는 2~3스텝으로 압축.
